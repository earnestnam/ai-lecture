# AI 활용 강의 웹사이트 구축 — 에이전트 설계서

> Claude Code에서 구현 시 참조하는 계획서

---

## 1. 작업 컨텍스트

### 배경 및 목적

회사 내 비개발자들이 AI 도구를 업무에 자연스럽게 활용할 수 있도록 돕는 **강의용 웹사이트**를 구축한다. 단순 도구 소개가 아니라, "왜 AI를 써야 하는가"부터 시작해 Claude·Gemini 생태계를 실전 수준까지 다루는 **자기 주도형 학습 가이드**를 목표로 한다.

### 참고 모델

[AI Roasting — Claude 완전 정복](https://airoasting.github.io/claude_guide/)의 구조·UX를 벤치마크한다. 메인 랜딩 페이지 + 개별 콘텐츠 페이지로 구성된 반응형 정적 사이트이며, 각 페이지 안에 읽기 콘텐츠·실습 과제·체크리스트 등이 포함된 형태.

### 범위

| 항목 | 설명 |
|------|------|
| **입력** | 커리큘럼 구조(이 설계서), 참고 사이트 디자인 패턴 |
| **출력** | GitHub Pages로 배포 가능한 정적 웹사이트 (HTML/CSS/JS) |
| **콘텐츠** | Claude Code가 초안 생성 → 사용자가 검수·수정 |
| **인터랙티브 요소** | AI 활용 수준 진단 퀴즈, 강의별 실습 과제 및 체크리스트 |

### 제약조건

- **정적 사이트 전용**: 서버 사이드 로직 없음. 모든 인터랙션은 클라이언트 JS로 처리
- **GitHub Pages 배포**: 빌드 도구 최소화 (Vanilla HTML/CSS/JS 또는 가벼운 SSG)
- **비개발자 대상**: 콘텐츠 톤은 친근하고 실용적. 기술 용어 최소화, 비유·시각 자료 적극 활용
- **반응형 필수**: 데스크탑 우선이되 모바일에서도 읽기 가능해야 함
- **콘텐츠 업데이트 용이**: 새 강의 페이지 추가 시 메인 페이지만 수정하면 되는 구조

### 용어 정의

| 용어 | 의미 |
|------|------|
| **랜딩 페이지** | 전체 커리큘럼을 카드 형태로 보여주는 메인 `index.html` |
| **콘텐츠 페이지** | 개별 강의 내용이 담긴 HTML 파일 (예: `ai-why.html`) |
| **섹션** | 랜딩 페이지에서 구분되는 학습 영역 (진단, 기본기, Claude, Gemini 등) |
| **카드** | 랜딩 페이지에서 각 콘텐츠 페이지로 연결되는 클릭 가능한 UI 블록 |

## 2. 커리큘럼 구조

### 전체 사이트맵

```
index.html (랜딩 페이지)
│
├── [섹션 0] 진단 — 내 수준 파악
│   ├── ai-why.html          왜 AI를 써야 하는가?
│   └── ai-level-quiz.html   나의 AI 활용 수준 진단 (인터랙티브 퀴즈)
│
├── [섹션 1] AI 기본기
│   ├── ai-landscape.html     AI 도구 지도 — 뭐가 있고 뭘 써야 하나
│   ├── prompt-basics.html    프롬프트 잘 쓰는 법
│   └── ai-work-habits.html   AI와 일하는 습관 만들기
│
├── [섹션 2] Claude 영역
│   ├── claude-intro.html         Claude 시작하기 (프로젝트 설정)
│   ├── claude-plugin.html        플러그인 — 크롬·파워포인트·엑셀
│   ├── claude-cowork.html        클로드 코워크 — 파일·드라이브 연결
│   └── claude-code.html          클로드 코드 — 바이브 코딩 입문
│
├── [섹션 3] Gemini 영역
│   ├── gemini-intro.html         Gemini 활용법
│   ├── notebooklm.html           NotebookLM 활용법
│   └── apps-script.html          Apps Script로 자동화하기
│
└── [부록]
    └── glossary.html             AI 용어 사전
```

### 각 콘텐츠 페이지 구성 패턴

모든 콘텐츠 페이지는 다음 구조를 공유한다:

```
[헤더: 사이트 로고 + 네비게이션]
[페이지 타이틀 + 예상 소요시간 + 난이도 배지]
[본문 콘텐츠]
  - 도입: 왜 이걸 배워야 하는가 (1~2문단)
  - 핵심 내용: 단계별 설명, 스크린샷 자리표시자, 팁 박스
  - 실습 과제 / 체크리스트 (해당 시)
[이전/다음 페이지 네비게이션]
[푸터]
```

## 3. 워크플로우 정의

### 전체 흐름

```
[1] 프로젝트 초기화
 │
 ▼
[2] 공통 템플릿·스타일 생성
 │
 ▼
[3] 랜딩 페이지 구축
 │
 ▼
[4] 콘텐츠 페이지 생성 (반복)
 │  ├─ 4a. 페이지 골격 생성 (템플릿 기반)
 │  ├─ 4b. 콘텐츠 초안 작성 (LLM)
 │  ├─ 4c. 인터랙티브 요소 구현 (해당 시)
 │  └─ 4d. 검증
 │
 ▼
[5] 통합 테스트 및 최종 검수
 │
 ▼
[6] GitHub Pages 배포
```

### 단계별 상세

#### [1] 프로젝트 초기화

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 |
| **작업** | 폴더 구조 생성, Git 초기화, `.nojekyll` 파일 생성 |
| **성공 기준** | 폴더 구조가 설계와 일치 (스키마 검증) |
| **검증 방법** | 스크립트로 필수 디렉토리 존재 확인 |
| **실패 시** | 자동 재시도 (최대 2회) |

#### [2] 공통 템플릿·스타일 생성

| 항목 | 내용 |
|------|------|
| **처리 주체** | LLM 생성 + 스크립트 검증 |
| **작업** | CSS 변수 정의, 반응형 레이아웃, 공통 헤더/푸터, 카드 컴포넌트, 콘텐츠 페이지 기본 레이아웃 |
| **성공 기준** | (1) 반응형 브레이크포인트 동작 (2) 참고 사이트와 유사한 시각적 느낌 (3) 다크/라이트 모드 대응 |
| **검증 방법** | LLM 자기 검증 — 생성된 CSS를 리뷰하여 브레이크포인트·변수 누락 확인 |
| **실패 시** | 자동 재시도 (최대 2회) |

**LLM 판단 영역**: 참고 사이트의 디자인 패턴을 분석하여 색상 팔레트, 타이포그래피, 카드 스타일 결정

#### [3] 랜딩 페이지 구축

| 항목 | 내용 |
|------|------|
| **처리 주체** | LLM 생성 |
| **작업** | 히어로 섹션, 섹션별 카드 그리드, 학습 로드맵 시각화, 반응형 네비게이션 |
| **성공 기준** | (1) 모든 섹션·카드가 커리큘럼과 일치 (2) 카드 링크가 올바른 파일을 가리킴 (3) 모바일에서 카드가 1열로 정렬 |
| **검증 방법** | 규칙 기반 — 링크 유효성 검사 스크립트 + LLM 자기 검증 (구조 리뷰) |
| **실패 시** | 자동 재시도 (최대 2회) |

**LLM 판단 영역**: 각 카드의 아이콘·이모지 선택, 설명 문구 작성, 시각적 구성 결정

#### [4] 콘텐츠 페이지 생성 (반복 단계)

이 단계는 사이트맵의 각 HTML 파일마다 반복 실행한다.

**4a. 페이지 골격 생성**

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 (템플릿 복사) |
| **작업** | 공통 템플릿에서 새 HTML 파일 생성, 메타데이터(제목, 난이도, 소요시간) 삽입 |
| **성공 기준** | HTML 파일 생성됨, 필수 메타 태그 존재 |
| **검증 방법** | 스키마 검증 |
| **실패 시** | 자동 재시도 (최대 2회) |

**4b. 콘텐츠 초안 작성**

| 항목 | 내용 |
|------|------|
| **처리 주체** | LLM (핵심 판단 단계) |
| **작업** | 페이지 주제에 맞는 본문 콘텐츠 생성. 도입부, 핵심 설명, 실습 과제, 팁 박스 포함 |
| **성공 기준** | (1) 비개발자가 이해할 수 있는 톤 (2) 최소 3개 이상의 구체적 예시 포함 (3) 실습 과제 또는 체크리스트 포함 |
| **검증 방법** | LLM 자기 검증 — 톤·완성도·누락 여부 체크 |
| **실패 시** | 자동 재시도 (최대 2회), 2회 실패 시 에스컬레이션 (사용자에게 방향 확인) |

**LLM 판단 영역**:
- 대상 독자(비개발자)에 맞는 설명 수준 조절
- 적절한 비유·사례 선택
- 실습 과제 난이도 설정
- 스크린샷 삽입 위치 결정 (자리표시자로 마킹)

**4c. 인터랙티브 요소 구현**

| 항목 | 내용 |
|------|------|
| **처리 주체** | LLM 설계 + 코드 생성 |
| **해당 페이지** | `ai-level-quiz.html` (진단 퀴즈), 각 페이지의 체크리스트 |
| **성공 기준** | (1) 퀴즈: 질문 표시 → 선택 → 결과 계산 → 추천 경로 표시 (2) 체크리스트: 체크 상태가 localStorage에 저장 |
| **검증 방법** | 규칙 기반 — JS 콘솔 에러 없음 확인 + LLM 자기 검증 (로직 리뷰) |
| **실패 시** | 자동 재시도 (최대 3회) |

**4d. 페이지 검증**

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 |
| **작업** | HTML 유효성, 링크 검증, 필수 구조 확인 |
| **검증 방법** | 규칙 기반 — 필수 섹션 존재, 이전/다음 링크 유효, 이미지 alt 텍스트 존재 |
| **실패 시** | 자동 재시도 (최대 2회) |

#### [5] 통합 테스트 및 최종 검수

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 + LLM + 사람 |
| **작업** | 전체 사이트 링크 검증, 반응형 테스트, 콘텐츠 일관성 확인 |
| **성공 기준** | (1) 깨진 링크 0개 (2) 모든 페이지에 네비게이션 존재 (3) 일관된 디자인 |
| **검증 방법** | 스크립트(링크 체크) + LLM 자기 검증(일관성) + 사람 검토(최종 품질) |
| **실패 시** | 에스컬레이션 — 발견된 이슈 목록을 사용자에게 제시 |

#### [6] GitHub Pages 배포

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 |
| **작업** | Git add/commit/push, GitHub Pages 설정 확인 |
| **성공 기준** | 배포 URL에서 사이트 접근 가능 |
| **검증 방법** | 규칙 기반 — HTTP 200 응답 |
| **실패 시** | 에스컬레이션 (Git/GitHub 설정 문제는 사람이 해결) |

### 분기 조건 요약

```
콘텐츠 페이지 생성 시:
  ├─ 인터랙티브 요소가 필요한 페이지인가?
  │   ├─ YES → 4c 실행 (퀴즈 또는 체크리스트 JS 구현)
  │   └─ NO  → 4c 스킵
  │
  └─ 콘텐츠 검증 통과?
      ├─ YES → 다음 페이지로
      └─ NO  → 재시도 (2회) → 실패 시 에스컬레이션
```

### 4.2 에이전트 구조: 단일 에이전트

**서브에이전트를 사용하지 않는 이유:**

- 워크플로우가 순차적이며 단순 (페이지를 하나씩 생성)
- 전체 지침이 CLAUDE.md 하나에 담길 수 있는 분량
- 도메인 지식이 동일 (비개발자 대상 AI 교육)
- 컨텍스트 윈도우 압박이 크지 않음 (각 페이지 독립적)

**대신 스킬로 역할을 분리한다:**

| 스킬 | 역할 | 트리거 |
|------|------|--------|
| `page-generator` | 템플릿 기반 새 HTML 파일 생성 | 새 콘텐츠 페이지 작업 시작 시 |
| `content-writer` | 비개발자 톤의 콘텐츠 초안 생성 | 빈 페이지에 콘텐츠를 채울 때 |
| `quiz-builder` | 진단 퀴즈·체크리스트 JS 구현 | 인터랙티브 요소가 필요한 페이지 |
| `site-validator` | 링크·구조·일관성 검증 | 페이지 완성 후, 또는 통합 테스트 시 |

### 4.3 CLAUDE.md 핵심 섹션 목록

CLAUDE.md에 포함될 섹션 (상세 내용은 구현 시 작성):

1. **프로젝트 개요** — 무엇을 만드는지, 대상자는 누구인지
2. **디자인 원칙** — 참고 사이트 패턴, 색상·타이포그래피 규칙, 반응형 기준
3. **커리큘럼 구조** — 섹션·페이지 목록과 순서
4. **콘텐츠 작성 규칙** — 톤, 길이, 필수 포함 요소 (도입·본문·실습·네비게이션)
5. **코딩 규칙** — HTML/CSS/JS 컨벤션, 파일 명명 규칙
6. **작업 순서** — 어떤 순서로 페이지를 만들 것인가
7. **검증 체크리스트** — 각 페이지 완성 시 확인할 항목
8. **스킬 참조** — 어떤 스킬을 언제 사용하는지

### 4.4 작업 단계별 처리 방식

| 단계 | 에이전트(LLM) 판단 | 스크립트 처리 |

| 스타일 시스템 | 색상·레이아웃 결정, CSS 코드 생성 | — |
| 랜딩 페이지 | 카드 문구·아이콘 선택, HTML 생성 | — |

| 콘텐츠 작성 | 본문 생성, 톤 조절, 예시 선택 | — |
| 퀴즈·체크리스트 | 질문 설계, 점수 로직 설계 | JS 코드 생성 |

### 4.5 스킬 상세

#### `page-generator`

| 항목 | 내용 |
|------|------|
| **역할** | 커리큘럼 메타데이터를 읽고, 공통 템플릿에서 새 HTML 파일을 생성 |
| **트리거** | "새 페이지 생성", "다음 페이지 작업" |
| **입력** | 페이지 ID (예: `claude-plugin`), `curriculum.json`의 메타데이터 |
| **출력** | `/src/pages/{page-id}.html` (골격만, 콘텐츠 비어있음) |
| **스크립트** | `create-page.sh` — sed로 템플릿의 플레이스홀더를 치환 |

#### `content-writer`

| 항목 | 내용 |
|------|------|
| **역할** | 빈 페이지 골격에 비개발자 톤의 콘텐츠를 채움 |
| **트리거** | 골격 생성 완료 후 자동 |
| **입력** | 골격 HTML 파일, `tone-guide.md`, 페이지 주제 |
| **출력** | 콘텐츠가 채워진 HTML 파일 |
| **참조** | `tone-guide.md` — 문체 규칙, 금지 표현, 예시 패턴 |

#### `quiz-builder`

| 항목 | 내용 |
|------|------|
| **역할** | 진단 퀴즈 UI·로직, 페이지별 체크리스트 구현 |
| **트리거** | `ai-level-quiz.html` 작업 시, 또는 체크리스트 필요 페이지 |
| **입력** | 퀴즈 질문 목록 (LLM이 생성), 체크리스트 항목 |
| **출력** | `quiz.js`, `checklist.js` 및 해당 HTML 마크업 |
| **참조** | `quiz-template.html` — 퀴즈 UI 뼈대 |

#### `site-validator`

| 항목 | 내용 |
|------|------|
| **역할** | 사이트 전체 또는 개별 페이지의 품질 검증 |
| **트리거** | 페이지 완성 시, "검증해줘", "테스트 돌려줘" |
| **입력** | 검사 대상 파일 경로 |
| **출력** | 검증 결과 리포트 (통과/실패 항목 목록) |
| **스크립트** | `check-links.sh` — grep으로 href 추출 후 파일 존재 확인, `check-structure.sh` — 필수 HTML 요소 존재 확인 |

### 4.6 주요 산출물 파일 형식

| 산출물 | 형식 | 위치 |
|--------|------|------|
| 랜딩 페이지 | HTML | `/src/index.html` |
| 콘텐츠 페이지 | HTML | `/src/pages/*.html` |
| 스타일시트 | CSS | `/src/css/style.css` |
| 인터랙티브 로직 | JS | `/src/js/*.js` |
| 커리큘럼 메타데이터 | JSON | `/.claude/skills/page-generator/references/curriculum.json` |
| 검증 리포트 | 텍스트 (stdout) | 콘솔 출력 |

### 4.7 데이터 전달 방식

이 프로젝트는 **단일 에이전트 + 파일 기반 전달**을 사용한다:

- 커리큘럼 구조 → `curriculum.json` 파일로 모든 스킬이 참조
- 페이지 골격 → HTML 파일로 `/src/pages/`에 저장, 이후 스킬이 읽고 수정
- 검증 결과 → stdout 출력 (파일 저장 불필요)
- 스킬 간 의존성 없음 — 모든 스킬은 CLAUDE.md(메인 에이전트)를 통해 순차 호출

## 5. 기술 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| **마크업** | Vanilla HTML5 | 빌드 도구 없이 GitHub Pages 즉시 배포 |
| **스타일** | Vanilla CSS (CSS 변수 활용) | 프레임워크 의존성 제거, 유지보수 용이 |
| **인터랙션** | Vanilla JS (ES6+) | 번들러 불필요, 브라우저 직접 실행 |
| **저장** | localStorage | 퀴즈 결과·체크리스트 진행 상태 저장 |
| **배포** | GitHub Pages | 무료, 참고 사이트와 동일 |
| **버전 관리** | Git + GitHub | 표준 |

**빌드 프로세스 없음**: `/src` 디렉토리를 GitHub Pages 루트로 직접 사용하거나, 단순 복사 스크립트로 `/output`에 배포.
